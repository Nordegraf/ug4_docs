SVN Basics

SVN is a version control system. That means it is a software to coordinate software development within a team.

SVN is installed on a server, and there are

  • one or more SVN Repositories which keep the source codes for a project
  • SVN Users, which can
  • Download the source code (checkout)
  • Upload/check in changes (commit)
  • Download changes others uploaded (update)

Every time a users successfully commits changes to the repository (that is: uploaded some changes), the revision number will be increased by one. The most up-to-date version is called the current revision, the current revision number, HEAD revision or just HEAD.

SVN Repositories

an SVN repository is a URL to a server, added with a description which repository from the server you want to have. For ug4, the SVN server is svn://fuji.gcsc.uni-frankfurt.de, and the repository there is called ug4, so the complete SVN URL is svn://fuji.gcsc.uni-frankfurt.de/ug4.

SVN Users

Sebastian Reiter can create a new SVN user for you. SVN Users consists of a username and a password.

Download the source code (checkout)

To checkout ug4's trunk directory to the directory ug4svn, we do

svn co svn://fuji.gcsc.uni-frankfurt.de/ug4/trunk ug4svn

SVN will prompt you for your password: 

Authentication realm: <svn://fuji.gcsc.uni-frankfurt.de:3690> fuji-svn-repositories
Password for 'mrupp':

after that, ug4 (HEAD revision) will be downloaded. See also Partial Checkout.

Note
All commands following are relative to the current directory, and working in a subdirectory will not effect the directories above.

Upload/check in changes (commit)

You can work now normally with the code, perform changes and so on. Say you changed the file ug4/plugins/experimental/myplugin/bla.cpp. Then

svn commit plugins/experimental/myplugin/bla.cpp -m"i added some usefull stuff"

Will upload or check in (commit) your changes to the server. Note that you have to specify some describing text via -m.

Download changes others uploaded (update)

Say i am another user, and i need the changes you made to ug4/plugins/experimental/myplugin/bla.cpp, then i can do a

svn update

in the ug4 directory, and i'll have the new version. You can see checkout as the initial update.

Note
When you do a svn update in a subdirectory, only the subdirectory will be updated.
SVN always keeps track of the last updated/checked out version (see diff and status).

Add, delete or rename files or directories

If you create a new file, say ug4/plugins/experimental/myplugin/blub.cpp, the file is not under version control or not versioned. That means, SVN won't upload changes of that file. To change that, do

svn add plugins/experimental/myplugin/blub.cpp
svn commit plugins/experimental/myplugin/blub.cpp -m"added blub.cpp"

Note that you also have to commit the changes (in this case: addition of a file). In the same way you can remove files (svn rm file) and rename files (svn rename fileOld fileNew), same is also true for directories.

Knowing what you have to check in

svn status

result will be for example 

?   not_under_version_control.txt
M file_i_modified.cpp
A file_i_added.cpp
D file_i_deleted.cpp
! missing_file.cpp

Files marked with '?' are not under version control. Files marked with '!' are files you deleted, but are under version control. They will be recreated when you do a svn update. Use svn rm theFile.cpp to delete files.

Note
All files marked with an M or A will be checked in if you to a svn commit -m"bla" (that is a commit of the current directory).

Reverting your changes

Sometimes you want to revert the changes you made (that way you can also be sure to be able to update):

svn revert filename.cpp

(also possible with directories, use with care and better check with svn status).

Conflicts

Now two people are working on plugins/experimental/myplugin/bla.cpp, so one day it happens that user A and B did some different changes bla.cpp, and now both want to commit the changes. By conincidence, user A is first, everything is ok. Now user B wants to commit his changes, but the error message is "file is out of date", user B has to do an update first. Now since the file bla.cpp was changed by user B, SVN has to merge the changes from the update process with the changes user B made. As in all synchronization problems, there can be conflicts, and various ways to solve them.

Conflict discovered in 'bla.cpp'.
Select: (p) postpone, (df) diff-full, (e) edit,
        (mc) mine-conflict, (tc) theirs-conflict,
        (s) show all options:

see here about how to resolve conflicts. If you want to throw away your changes, and just use the version from the server, enter tf (theirs full). If you made some changes you want to keep, use tc (theirs conflict).

SVN remembers the state of the last version you checked out (or updated), so it always knows which files you changed and which changes you made.

Error, other conflicts, troubleshooting

svn: Failed to add file 'bla.cpp': an unversioned file of the same name already exists

SVN is afraid to overwrite the file bla.cpp, it is an unversioned file. You can rename the locally created bla.cpp to my_bla.cpp, and then the update should work. After that, merge your changes from my_bla.cpp manually.

svn: Failed to add directory 'another/directory': an unversioned directory of the same name already exists

Again, SVN is afraid to overwrite the directory another/directory. Rename it to another/directory_my, update, and merge the two directories manually.

Incompatible SVN Versions

Sometimes the SVN version installed on the cluster is not compatible with yours, so when you used your local svn on the remote directory, you can't use the remote svn anymore. See this script to Change SVN format.

SVN/SSH Tunneling

On some clusters it is forbidden to connect directly to the internet. That also includes fuji.gcsc.uni-frankfurt.de. On those clusters, you can not check out or update svn sources. The cure for this problem is called SSH Tunneling. Say you connect to your cluster with ssh marti.nosp@m.n@cl.nosp@m.uster.nosp@m..com. You now need to replace this call with

ssh -R5337:fuji.gcsc.uni-frankfurt.de:3690 martin@cluster.com

Here the -R specifies that all connections to localhost:5337 are forwarded to fuji.gcsc.uni-frankfurt.de:3690. It is important to issue this ssh command on a computer which can connect to fuji.gcsc.uni-frankfurt.de. Now we instead of svn co svn://fuji.gcsc.uni-frankfurt.de/ug4/trunk we can do

svn co svn://localhost:5337/ug4/trunk

You can now do a svn co or update. However, some externals might not be able to be checked out.

cd externals
svn propedit svn:externals .

(If you are getting an error that no editor is set use export EDITOR=emacs or similar) Now change

boost_1_56_0/boost svn://fuji.gcsc.uni-frankfurt.de/boost_mirror/boost_1_56_0/boost

to

boost_1_56_0/boost svn://localhost:5337/boost_mirror/boost_1_56_0/boost

Now you should be able to do a complete update/co.

For convenience it is nice to have an alias for such clusters like

alias ssh-cluster="ssh -Y-R5337:fuji.gcsc.uni-frankfurt.de:3690 martin@cluster.com"

since you need to do this ssh -R every time you want to do svn on the cluster.

Note
uginstall also supports this, e.g.
uginstall SuperLU -svnServer localhost:5337
When you are doing SVN/SSH Tunneling, all your svn data is transfered through your connection. To avoid this, you might want to first connect to a server with good connection (like quadruped at GCSC), and connect from there with tunneling to the cluster.

More

SVN Revision number

Getting the svn revision number of this directory and the subdirectories

svnversion

The output has the form 

LOWESTREVISION[:HIGHESTREVISION][M]

So 9476:9496M means the lowest revision number is 9476, the highest is 9496, and you modified some files which are under version control (hence the 'M'). An output like 9476 means that all files are on revision number 9476 and non of them is modified.

ug4 or ug4's revision number

With "updating ug4" we mean you need to update all of ug4, not only some subdirectory. For that, go to your ug4 base directory, and enter svn update. There you can also do a svnversion to get the SVN version of whole ug4.

Other Usernames

Without specifying the username explicitly, SVN assumes that the SVN username is the name of the current user. If SVN has the wrong username, you might have your checkout line to

svn co --username horst svn://fuji.gcsc.uni-frankfurt.de/ug4/trunk ug4svn
Note
SVN remembers your user names. This can lead to errors, since other updates might then also use that user. Users are saved in ~/.subversion/auth, so when you want to remove user 'otherUser' you could try to delete all files listed in a fgrep -r otherUser ~/.subversion/auth/ .

Log or History

Showing the SVN log or history

svn log bla.cpp

will show the complete log entries of bla.cpp.

svn log -l 7

will show the last 7 entries of the log of the current directory (and its subdirectories and files).

diff

Knowing what you changed in the files (diff)

svn diff bla.cpp

will compare the file bla.cpp to the version you last updated.

Other Examples

current directory is CURDIR

  • svn co -r 9767 svn://fuji.gcsc.uni-frankfurt.de/ug4/trunk ug4svn
    checking out revision number 9767 (instead of HEAD)
  • svn status
    show svn status (modified, versioned, added, deleted) of CURDIR
  • svn status mydir
    show svn status of CURDIR/mydir (if mydir relative) or mydir (if mydir absolute)
  • svn commit -m"commiting CURDIR"
    upload changes in CURDIR and its subdirectories
  • svn commit myfile.cpp -m"commiting CURDIR"
    upload changes in myfile.cpp
  • svn commit bla1.cpp bla2.cpp -m"commiting CURDIR"
    upload changes in bla1.cpp and bla2.cpp
  • svn add myfile.cpp
    add the file myfile.cpp to the versioning system. needs commit to upload changes.
  • svn add myfile.cpp myfile2.cpp
    adding the files myfile.cpp and myfile2.cpp
  • svn add *.cpp
    adding all .cpp files in CURDIR
  • svn add subdir subdir/*.cpp
    adding CURDIR/subdir and all .cpp files in CURDIR/subdir
  • svn update -r 1978 bla.cpp
    get the bla.cpp version from revision 1978 (used e.g. when you need to find out bugs which didn't happened some revisions ago)
  • svn update -r {2013-02-03} mydir
    get the version from 02.03.2013 (date) of CURDIR/mydir
  • svn update -r PREV
    get the previous version of CURDIR
  • svn update
    update CURDIR
  • svn update SUBDIR
    update the contents of CURDIR/SUBDIR
  • svn diff
    show diff of CURDIR, subdirectories, and files
  • svn diff bla.cpp
    show diff of bla.cpp
  • svn diff -r HEAD bla.cpp
    compare bla.cpp with the current HEAD revision on the server
  • svn revert bla.cpp
    revert the changes you've made to bla.cpp
  • svn revert -R .
    revert the changes you've made to this directory and the subdirectories

SVN in Eclipse

Eclipse provides an easy-to-use interface for SVN in Eclipse. If you have code which is under version control, you will see the revision number on the right side of the files. On the left side you can see a '>', that means this file is modified, or, on directories: this directory or a subdirectory contains modifications.

Right mouseclick on a file, you have

  • Team:
    • Commit : performs a svn commit of this file/directory. You can enter a commit message, you see which changes are committed and you can deselect some of the files.
      Warning
      Unlike svn commit, if you have unversioned files in your directory, they will be added. You have to deselect them again. Consider adding those files to svn:ignore.
    • Update : svn update of this file
    • Update to revision: bring this file to an older revision state (used e.g. when you need to find out bugs which didn't happened some revisions ago)
    • Revert: svn revert of this file/directory.
    • Show History: shows the svn log of this file
    • Add to svn:ignore : This way, svn won't care about these files anymore. They also won't show up on svn status, and Eclipse won't add them if you commit their parent directory.
  • Compare With... : various ways to compare this file
  • Replace With... : various ways to replace this file
Note
You disable that new files are automatically added when using Commit here: Preferences -> Team -> SVN -> Uncheck "Check the new resources in the commit dialog".

Some Warnings

  • Warning
    Do not add unnecessary files. That means: Make sure you don't add unnecessary files. In ug4, a click on ug4's base directory and there a commit will almost always add lots of unnecessary files like your build, bin or lib files. Make use of the svn:ignore option or that those files are unchecked.
  • Warning
    Do not add lots of result files or huge data files. Keep in mind everyone has to download them.

SVN Troubleshooting

Sometimes SVN is not getting it right. Some problems are:

  • svn update says 'Skipped .'. That is because your current directory is not under version control. If you do a 'ls -la', you will see you don't have a .svn folder there. That can have several reasons:
    • You are in a directory which is not meant to be in SVN, for example the subfolders bin, debug or release, . They are not under version control because those files are generated from the build script, and contain a lot of big files, and we don't want everyone to be forced to download unnecessary files. Solution: Change to another directory.
    • Sometimes you are sure that a directory should be in svn, but is not, because .svn was deleted accidentially. Then: say your directory is THEDIR/THESUBDIR and you are in THEDIR/THESUBDIR. then
      cd ..
      mv THESUBDIR THESUBDIR_mine
      svn update THESUBDIR
      This way, you save your directories contents, and forced SVN to re-download the contents of THESUBDIR, which then contains a .svn again. If you made changes to THESUBDIR, use
      diff THESUBDIR THESUBDIR_mine
      Now you can manually re-insert your changes from THESUBDIR_mine to THESUBDIR.
    • If the same happended to you in the base directory of ug4, you have to rename your whole ug4 to ug4_mine and re-checkout ug4.
  • I accidentially deleted directory THESUBDIR which was under SVN control. How to get it back?
    cd ..
    svn update THESUBDIR
  • svn update / co takes a while, then displays some connection error.
    This might be a problem on a cluster that you are not allowed to connect to the svn server (see SVN/SSH Tunneling), or our svn server is down.

Partial Checkout

In most cases it is not necessary to check out the whole ug4 repository. Therefore SVN provides the possibility to only checkout certain directories of a repository.

To achieve this, first checkout the trunk of ug4 without recursing into subdirectories by

svn co --depth immediates svn://fuji.gcsc.uni-frankfurt.de/ug4/trunk ug4svn

Now only folders and files of ug4/trunk without their content is downloaded. To get a minimal compiling copy of ug4 issue the following commands:

cd ug4svn
cd ugbase && svn update --set-depth infinity && cd ..
cd cmake && svn update --set-depth infinity && cd ..
cd scripts && svn update --set-depth infinity && cd ..

This will download all the core parts of ug4 (ugbase), the various files and scripts for CMake (cmake) and the utility scripts, which might turn out helpful (scripts).

If you are working on a certain app without bothering about all the others, first get the list of all apps and then only check out your awesome one:

cd apps && svn update --set-depth immediates && cd ..
cd apps/your_awesome_app && svn update --set-depth infinity && cd ../..

Similar you proceed with plugins:

cd plugins && svn update --set-depth immediates && cd ..
cd plugins/core && svn update --set-depth immediates && cd ../..
cd plugins/experimental && svn update --set-depth immediates && cd ../..

Finally there are some great news for you: SVN will keep track and remember your settings regarding the depth. I.e. if you svn update in the root (i.e. ug4svn) it will only checkout and update those directories and files, which you have checked out so you will not get annoyed by a flood of changes in a boring app you don't care about!
